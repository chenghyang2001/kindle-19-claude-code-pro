# 練習 03 — 綜合挑戰：TDD 實戰

> 選配：完成前兩個練習後再挑戰

## 挑戰情境
用 TDD 方式實作一個「Email 格式驗證器」，體驗 Red-Green-Refactor 節奏。

## 規則
- 必須「先寫測試，再寫實作」
- 每個測試通過後才能繼續加下一個測試
- 記錄每個階段的時間

## 要求

### 階段一：Red（寫測試，讓它失敗）
先寫以下 5 個測試（不實作任何邏輯，確認測試全部 FAIL）：

1. 有效 email（`user@example.com`）→ 回傳 True
2. 無 @ 符號（`userexample.com`）→ 回傳 False
3. 多個 @（`user@@example.com`）→ 回傳 False
4. 空字串（`""`）→ 回傳 False
5. 特殊字元在 local part（`user+tag@example.com`）→ 回傳 True（這是合法的！）

記錄時間：_____ 分鐘

### 階段二：Green（最小實作讓測試通過）
用最少的程式碼讓 5 個測試全部通過。
不要過度設計，只要通過就好。

記錄時間：_____ 分鐘

### 階段三：Refactor（改善程式碼品質）
在測試全部通過的前提下：
1. 加入適當的型別提示
2. 加入 docstring
3. 簡化邏輯（如果有重複或不必要的複雜度）

確認測試仍然全部通過。

記錄時間：_____ 分鐘

## 反思問題
完成後回答：
1. TDD 的哪個階段最困難？為什麼？
2. 你在 Refactor 階段改了什麼？這些改動在「先實作後測試」時會自然發生嗎？
3. 你覺得 TDD 適合哪類型的工作？不適合哪類型？
