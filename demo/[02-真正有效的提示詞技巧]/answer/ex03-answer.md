# 練習 03 解答：個人提示詞庫

**建立日期：** 2026-06-17

## 提示詞庫（5 個）

---

### /debug
**使用時機：** 有 error message、stack trace 或可描述的異常行為時

**模板：**
```
你是一個資深除錯工程師。
錯誤訊息：{貼上 error message 或 stack trace}
發生位置：{檔案名稱:行數}
預期行為：{描述正確應該發生什麼}
請找出根本原因並修復。
輸出：修復後的程式碼 + 一行根因說明。
```

**版本記錄：**
- 2026-06-17 v1：初版建立

---

### /review
**使用時機：** 功能完成、PR 前、或懷疑有隱藏問題

**模板：**
```
你是一個嚴格的 Code Reviewer。
請審查以下程式碼，優先找：邏輯錯誤、安全漏洞、效能問題。
輸出：條列式問題清單，每項附修改建議，按嚴重度排序（高/中/低）。
```

**版本記錄：**
- 2026-06-17 v1：初版建立

---

### /doc
**使用時機：** 函式或模組需要說明文件，或準備 PR 說明

**模板：**
```
你是一個技術文件工程師。
為以下程式碼產生說明文件，包含：
- 功能說明（一句話）
- 參數說明（名稱、型別、用途）
- 回傳值說明
- 使用範例（含輸入輸出）
格式：{Python docstring / JSDoc / Markdown}
```

**版本記錄：**
- 2026-06-17 v1：初版建立

---

### /refactor
**使用時機：** 程式碼可運作但可讀性差、函式超過 30 行、有重複邏輯

**模板：**
```
你是一個重構專家。
重構以下程式碼，目標：提升可讀性、消除重複邏輯。
限制：不改變輸入輸出介面、不引入新的依賴、保持相同行為。
輸出：重構後完整程式碼 + 改動說明（列出每項變更原因）。
```

**版本記錄：**
- 2026-06-17 v1：初版建立

---

### /api-design
**使用時機：** 設計新的 API 端點或評估現有 API 結構

**模板：**
```
你是一個 API 設計師，核心原則是向後相容。
設計以下功能的 API：{需求描述}
限制：RESTful 規範、JWT 認證、JSON 回傳、錯誤碼遵循 RFC 7807。
輸出：
1. 端點清單（METHOD /path）
2. 每個端點的 request/response 範例
3. 需要注意的 breaking change 風險
```

**版本記錄：**
- 2026-06-17 v1：初版建立

---

## 核心洞察

**「使用時機」比模板更重要。** 知道「什麼時候不該用 /debug，而該用 /review」，才是提示詞庫真正的價值所在。

**「限制」是提示詞最關鍵的元素。** 限制縮小 Claude 的猜測空間，讓輸出從「可能對」變成「穩定對」。
