# 練習 01 解答：重寫低效提示詞

## 三個低效提示詞診斷

### 1. 「幫我改一下這個 function」

**缺少的資訊：**
- 哪個 function（沒有指定目標）
- 怎麼改、改什麼目的（優化？修 bug？加功能？重構？）
- 什麼程式語言
- 輸出格式（完整程式碼？diff？要不要解釋？）

**可能誤解方向：** 隨意選一個函式、猜測修改方向、輸出格式不符預期

**改寫版：**
> 「你是一個 Python 後端工程師。請優化 `utils.py` 第 42 行的 `calculate_average()` 函式，目標是把時間複雜度從 O(n²) 降到 O(n)。限制：不改動函式的輸入輸出介面。輸出：只給修改後的函式程式碼，附一行註解說明改動原因。」

---

### 2. 「這個 bug 怎麼修」

**缺少的資訊：**
- 哪個 bug（沒有指定目標）
- bug 的症狀或錯誤訊息（最致命的缺失）
- 預期行為 vs 實際行為
- 重現條件

**額外致命問題：** 沒有提供 error message 或 stack trace。沒有症狀就無法診斷根本原因。

**改寫版：**
> 「你是一個 Python 除錯專家。`app/routes.py` 第 87 行的 `get_user()` 函式在傳入 `user_id=None` 時丟出 `AttributeError: 'NoneType' object has no attribute 'email'`。預期行為：應該回傳 404 而不是 crash。請找出根本原因並修復，輸出修正後的函式 + 一行解釋。」

---

### 3. 「寫個 API」

**四大元素全缺：**

| 元素 | 缺少的內容 |
|------|-----------|
| 角色 | 不知道從哪個角度設計（前後端？架構師？） |
| 任務 | API 做什麼？給誰用？什麼業務邏輯？ |
| 限制 | 框架、語言、認證方式、回傳格式都未指定 |
| 輸出格式 | 要 code？spec？文件？Postman collection？ |

**改寫版：**
> 「你是一個 Python 後端工程師。請為使用者管理系統設計 RESTful API，使用 FastAPI 框架。限制：所有端點需加 JWT 認證，回傳 JSON 格式，錯誤碼遵循 RFC 7807。輸出：Python 程式碼 + 每個端點的簡短說明。」

---

## 核心心法

```
模糊提示詞 → Claude 猜 → 輸出不穩定
具體提示詞 → Claude 執行 → 輸出可預測
```

**「限制」是最被低估的元素。** 加限制不是限制 Claude 的能力，而是縮小它的猜測空間。
