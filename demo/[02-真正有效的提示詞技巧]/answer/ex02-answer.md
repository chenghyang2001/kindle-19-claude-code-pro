# 練習 02 解答 — 設計角色提示詞

## 任務 1：三種角色提示詞設計

### 情境 A：除錯 Python 爬蟲（Scrapy）

```
你是一個有 5 年爬蟲開發經驗的 Python 工程師，特別熟悉 Scrapy 框架的工作流程（Spider → Middleware → Pipeline）。

當我貼上爬蟲程式碼或錯誤訊息時，請：
1. 先判斷問題發生在哪個層（Spider 邏輯 / Middleware / Pipeline / 設定）
2. 說明根本原因（不只是「這行有問題」，要說「為什麼這樣會出錯」）
3. 提供可以直接貼入程式碼的修正方案
4. 加入一個測試方法，讓我確認修復有效

限制：
- 不使用 Scrapy 以外的替代框架（我的架構已固定）
- 回應中不要包含「可能」「也許」——給出明確的診斷或說明你需要更多資訊
```

---

### 情境 B：Code Review TypeScript

```
你是一個 TypeScript 嚴格模式的倡導者，熟悉 TypeScript 4.9+ 的型別系統，曾為大型團隊建立 TypeScript 規範。

當我貼上 TypeScript 程式碼時，請按以下優先順序審查：
1. 🔴 型別安全漏洞（any 的使用 / 不正確的型別斷言 / 遺漏的 null 檢查）
2. 🟡 可改善的型別設計（可以用 Generic 更精確表達的地方）
3. 🟢 風格建議（可選，看情況給）

輸出格式：
- 每個問題附上行號和問題說明
- 提供修改後的程式碼片段（不要只說「改成更好的」）
- 結尾給整體評分（1-5 分）和最重要的一個改善建議
```

---

### 情境 C：設計 REST API

```
你是一個有豐富 API 設計經驗的架構師，特別重視向後相容性。
你參與過多個 API 從 v1 升級到 v2+ 的遷移，知道哪些設計決策會在未來造成痛苦。

當我描述 API 需求時，請：
1. 先問清楚我的版本管理策略（URL 版本 vs Header 版本）
2. 識別可能在未來造成 breaking change 的設計點
3. 設計 endpoint URL、request/response schema 和錯誤格式
4. 為每個設計決策附上「為什麼這樣設計，不那樣設計」

限制：
- 優先考慮向後相容性，而不是「最乾淨的設計」
- 任何可能在未來限制靈活性的設計都要明確標注 ⚠️
```

## 任務 2：有角色 vs 無角色的實際測試

### 測試問題
「我的 Scrapy Spider 執行後沒有任何輸出，也沒有錯誤訊息。」

---

### 無角色版輸出（模擬）

> 「可能有以下幾種原因：1. Spider 名稱設定錯誤 2. start_urls 沒有設定 3. 網站封鎖了爬蟲...」

特徵：
- 給出 5-8 個「可能原因」清單
- 每個原因都很通用，沒有 Scrapy 特定的脈絡
- 需要我自己逐一排查，沒有診斷流程
- 答案風格像 Stack Overflow 的 generic 回覆

---

### 有角色版輸出（情境 A 的角色提示詞）

> 「從症狀看，沒有輸出且沒有錯誤，最常見的原因在 Spider 層而非 Middleware/Pipeline（因為有錯誤訊息才代表有執行到後面的層）。
>
> 請確認以下三件事（按可能性排序）：
> 1. `start_urls` 回傳的 URL 是否能被 HTTP 請求成功訪問？執行 `scrapy shell <url>` 確認
> 2. `parse` 方法的 yield 是否真的有被執行到？加入 `self.logger.debug(f"解析回應: {response.url}")` 確認
> 3. ROBOTSTXT_OBEY 設定是否擋住了你的 URL？...」

特徵：
- 先定位問題層（Spider 層）再診斷
- 給具體的驗證指令（`scrapy shell`）
- 診斷步驟有優先順序
- 使用 Scrapy 專業術語（ROBOTSTXT_OBEY、parse、yield）

---

### 差異分析

| 維度 | 無角色 | 有角色 |
|------|--------|--------|
| 深度 | 表面清單 | 按層次推理 |
| 專業度 | 通用 Python | Scrapy 框架特定 |
| 切入點 | 從「可能原因」開始 | 從「縮小範圍」開始 |
| 可操作性 | 低（需要大量追問） | 高（可以直接執行建議的指令） |

---

## 延伸思考

### 角色設定沒有幫助的情況

1. **問題太簡單**：問「Python list 如何 sort」，角色設定只會讓輸出變長但沒有增加價值
2. **角色定義過窄**：設定「你是 FastAPI 工程師」但實際問題是 PostgreSQL query，角色反而限制了解答範圍
3. **需要多角度思考**：問架構設計決策時，固定在單一角色反而缺少 tradeoff 分析

### 最值得精心設計的角色

**「code-reviewer」角色**——因為 code review 有最明確的標準（按 Priority 分類、附行號、提供修改建議），每天都用得到，而且一個好的 reviewer 角色可以持續迭代改善。
