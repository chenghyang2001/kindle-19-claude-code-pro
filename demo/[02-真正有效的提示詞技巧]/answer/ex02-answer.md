# 練習 02 解答：設計角色提示詞

## 三個情境的角色設計

### 情境 A：除錯 Python 爬蟲

**角色提示詞：**
> 「你是一個資深 Python 工程師，專精 Scrapy 框架，曾處理過反爬蟲、動態渲染、編碼亂碼、pipeline 阻塞等問題。除錯時優先檢查 middleware 設定和 response 狀態碼。」

**為何有效：** 「資深」+「Scrapy 專精」+「具體踩坑經驗」三層疊加，讓 Claude 調用 Scrapy 特有的除錯知識子集。

---

### 情境 B：Code Review TypeScript

**角色提示詞：**
> 「你是一個嚴格的 TypeScript 專家，對型別安全有潔癖。Review 時優先找：`any` 的濫用、未處理的 null/undefined、不完整的 union type、interface 和 type 的誤用。」

**為何有效：** 明確指出優先找的問題類型，避免 Claude 做表面 review 而忽略型別安全。

---

### 情境 C：設計 REST API

**角色提示詞：**
> 「你是一個 API 設計師，核心原則是向後相容。設計時優先考量版本管理（/v1/、/v2/）、breaking change 的影響範圍、deprecation 策略。」

**為何有效：** 強調「向後相容」這個核心原則，讓 Claude 在每個設計決策都以此為優先考量。

---

## 角色設定的失效情境

角色設定在以下情況沒有幫助，甚至反效果：

1. **任務太簡單**：「把 list 排序」加角色只是浪費 token
2. **角色與任務不匹配**：叫「前端工程師」去優化 SQL，視角偏差
3. **角色描述太長**：佔用 context window，排擠實際程式碼

**甜蜜點：** 角色描述 1-3 句話，超過報酬遞減。

---

## 角色設定使用判斷

```
任務 < 5 行改動   → 不需要角色，直接下指令
任務涉及特定框架   → 加精準角色（1-2 句）
任務跨多個領域    → 不要一個角色塞所有，分開提問
```
