重構執行計畫（前 3）

#1：消除 notion_creator 雙實作重複（得分 2.00）

  • 手法：Extract 共用 payload 建構為 _build_notion_payload()
  • 先加測試：mock API / subprocess 回傳各一個
  • 預估時間：30 分鐘
  • 預期改善：減少 40 行重複碼，改 payload 格式只需改一處

#2：刪除重複 fetch 邏輯（得分 2.00）

  • 手法：Remove Duplicate，直接呼叫 gmail_fetcher
  • 先加測試：確認兩者回傳格式一致
  • 預估時間：20 分鐘
  • 預期改善：修 Gmail bug 只需改一個地方

#3：拆解 run_briefing.main 106 行（得分 1.67）

  • 手法：Extract Function → fetch_stage / filter_stage / summarize_stage / notify_stage
  • 先加測試：整合測試覆蓋完整 pipeline
  • 預估時間：60 分鐘
  • 預期改善：每個 stage 可獨立測試、替換

風險評估

最危險：拆解 run_briefing.main。原始碼有隱性狀態共享（cache、errors 清單在 main 內部流轉）。提取子函式時若漏傳這些狀態，會靜默失敗（不拋例外，只是不 cache）。

降低方式：先加端對端測試涵蓋 cache 行為，每提取一個 stage 就立刻跑全測試，不要一次全部提取。
