練習 02：程式碼品質 Checklist 實戰

10 項品質 Checklist（按危險程度排序）

優先項目如何驗證違反後果
🔴 1所有 SQL 用參數化查詢grep execute(f" 應零結果SQL 注入，DB 被讀取/刪除
🔴 2API Key / 密碼從環境變數讀取grep password= 只允許 os.environcommit 後永久洩漏
🔴 3所有外部 I/O 有 try/except找每個 connect() / requests.get() 確認有 try外部失敗讓程式 crash
🟠 4DB 連線必定關閉找每個 conn= 確認有 finally 或 with連線池耗盡，服務掛掉
🟠 5使用者輸入有驗證看函式入口的型別/範圍/長度檢查崩潰或被利用做 DoS
🟠 6敏感資訊不寫進 log搜尋 logger 輸出無 password/token 欄位log 系統洩漏機密
🟡 7函式有明確回傳型別標注看函式簽名有無 -> dict | None 等呼叫方不知要判斷 None
🟡 8不用裸 except:grep "except:" 應零結果吞掉 KeyboardInterrupt 等
🟡 9迴圈內不重複建立 DB 連線看迴圈裡有無 connect()N+1 連線，效能惡化
🟢 10有冒煙測試python file.py 直接跑一次基本功能無任何驗證

套用到 fetch_user 原始版本

#項目結果說明
1參數化查詢❌f-string 拼接 SQL
2環境變數❌password='1234' 硬編碼
3try/except❌完全沒有
4連線關閉❌無 finally，無 with
5輸入驗證❌user_id 未驗證型別
6log 無機密✅ 不適用完全沒有 log
7回傳型別❌無型別標注
8具體例外❌沒有任何 except
9迴圈連線✅只建一次連線
10冒煙測試❌無任何測試

讓 Checklist「不得不做」

層級工具觸發時機
本機即時Claude Code PostToolUse hook → ruff / bandit每次 Write .py 自動掃
commit 前pre-commit hook → bandit -r .git commit 時強制通過
CI 遠端GitHub Actions → bandit + pytestpush 後自動跑，失敗不讓 merge

結論：Checklist 是「知道要查什麼」，hook + CI 才是「保證有查」。人的記憶不可靠，exit code 永遠可靠。
