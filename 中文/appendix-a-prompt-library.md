# 附錄 A：快速參考提示詞庫

日常使用的快速參考資料。

使用前請將每個 `[括號項目]` 替換為您自己的值。

---

## A.1 五要素提示詞結構

```
1. 角色（ROLE）：     扮演資深 [語言/框架] 開發者。
2. 背景（CONTEXT）：  這是一個使用 [技術堆疊] 的 [框架] 專案。
                     相關檔案為 [檔名]。
3. 任務（TASK）：     [具體的命令式任務描述]。
4. 限制（CONSTRAINTS）：[具體的隱含需求/模式]。
5. 成功標準（SUCCESS）：[預期輸出格式 / 成功條件]
```

## A.2 功能鷹架

```
為 [功能名稱及描述] 建立鷹架。
依序建立檔案：schema → 查詢 → 服務動作。
接著元件 → 測試。遵循所有 CLAUDE.md 與程式碼慣例。
在撰寫任何程式碼前，先給我檔案清單。
```

## A.3 測試生成

```
為 [檔名] 撰寫完整測試套件。
涵蓋：正常路徑、所有錯誤案例、null/空值邊界情況。
測試行為，而非實作細節。目標覆蓋率 95% 以上。
```

## A.4 除錯流程

```
> 除錯此錯誤：[貼上完整錯誤訊息 + 堆疊追蹤]。
第一步：根本原因分析。
第二步：修復方案。
第三步：能攔截此 bug 的測試案例。
第四步：能攔截此 bug 的測試案例（確認）。
```

## A.6 PR 審查（`/review-pr`）

```
審查此分支與 main 的 git diff。
檢查：(1) 正確性 — 邊界情況，(2) 資安 — 輸入、
驗證、錯誤訊息，(3) CLAUDE.md 慣例，
(4) 效能（找 N+1 查詢）。
給我結論——不要單獨開 PR。
請勿直接 push 至 main 或修改正式環境資源。
```

## A.8 MCP 多工具工作流程

```
實作 [GitHub issue #N] 的功能。
1. 讀取 issue（GitHub MCP）。
2. 讀取相關資料庫 schema（Dialogue MCP）。
3. 確認當前最佳實踐（網路搜尋 MCP）。
4. 提出實作計畫。等待核准。
5. 核准後實作。
```
